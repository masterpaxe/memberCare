# MemberCare Database (PostgreSQL)

This folder contains a production-oriented relational schema derived from the SRS.

## Files

- `01_schema.sql`: Core DDL (tables, constraints, indexes)
- `02_seed.sql`: Branches, roles, permissions, basic role-permission mappings
- `03_views.sql`: Reporting views for dashboards and leadership summaries

## Execution Order

Run scripts in this order:

1. `01_schema.sql`
2. `02_seed.sql`
3. `03_views.sql`

Example using `psql`:

```bash
psql -U postgres -d membercare -f database/postgresql/01_schema.sql
psql -U postgres -d membercare -f database/postgresql/02_seed.sql
psql -U postgres -d membercare -f database/postgresql/03_views.sql
```

## Design Notes

- IDs use UUID values generated by `gen_random_uuid()` from `pgcrypto`.
- Sensitive workflows are represented using `pastoral_care_records`, `follow_up_records`, and `audit_logs`.
- Multi-branch access is enforced by `branch_id` relationships in operational tables.
- Contribution tracking is included as a controlled module in `contributions`.

## Immediate Backend Mapping

Suggested API aggregate boundaries:

- Identity and Access: users, roles, permissions
- Membership: members, dependents, status history
- Assimilation: visitors, new converts, follow-up records
- Attendance: sessions and records
- Care: pastoral care records
- Operations: departments, fellowships, classes, events
- Communications and Alerts: message logs, notifications
- Governance: audit logs

## Data Governance Baseline

- Enable backup strategy and PITR (point-in-time recovery).
- Restrict direct table access; use role-based app access.
- Encrypt data in transit and at rest.
- Add row-level security policies per branch when backend auth is in place.
